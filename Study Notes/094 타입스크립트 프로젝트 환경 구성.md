## 📝 핵심 개념 요약

- TS 프로젝트는 **npm 초기 세팅 → TS 설치 → tsconfig 설정** 순서로 구성
- `tsc`는 TS → JS로 컴파일
- `ts-node`는 JS 파일 생성 없이 TS를 즉시 실행
- tsconfig로 **타입 검사 옵션, 컴파일 대상, 모듈 시스템, JS 버전** 등을 지정
- target 변경 시 **TS 최신 문법이 과거 JS 문법으로 변환되는 과정 확인 가능**
- module을 변경하면 **commonjs(require) ↔ ES module(import/export)** 차이가 발생

## 📊 핵심 요약 표

| 개념 | 설명 |
| --- | --- |
| tsc | TS 파일을 JS로 변환 (컴파일러) |
| ts-node | 컴파일 + 실행을 한 번에 처리 (JS 생성 없음) |
| tsconfig.json | 컴파일 옵션 및 소스/출력 경로 설정 파일 |
| target | 어떤 버전의 JS로 변환할지 지정 |
| module | 어떤 모듈 시스템(CommonJS/ESM)을 사용할지 지정 |
| dist / src | dist: 컴파일 결과물, src: TS 소스 |

### 💻 실습 내용 정리

### 1) 프로젝트 생성 및 초기 환경 구성

```bash
mkdir topic-typescript-basic
cd topic-typescript-basic
npm init -y
npm i -g typescript
npm i -D typescript
npx tsc --init
```

### 2) package.json script 설정

```json
"scripts": {
  "build": "tsc",
  "start": "node main.js"
}
```

### 3) main.ts 생성

```tsx
console.log("타입스크립트 시작!");
```

### 4) 컴파일 & 실행

```bash
tsc main.ts   # main.js 생성
node main.js
```

    "module": "commonjs",
    "outDir": "./dist",

  "exclude": []

### ❗ 헷갈렸던 점 / 문제 해결:

### 헷갈렸던 점

- target과 module의 차이가 애매했음
- ES 버전별 차이를 외우기 어려웠음
- 왜 optional chaining이 오래된 JS 버전에서 사라지는지 이해가 어려웠음

### 문제 해결

- target은 “JS 버전”, module은 “모듈 시스템”이라는 개념으로 나누어 이해
- ES 버전 표를 통해 각 버전의 핵심 기능을 정리
- TSC가 최신 문법을 구 JS 문법으로 **트랜스파일**하는 과정으로 이해됨
- 실제 컴파일 코드 비교하며 동작 원리 체득

### 💡 느낀 점 / 배운 점:

- TypeScript 환경을 구성하는 과정이 단순한 설치 작업이 아니라,
    
    **빌드 시스템을 구성하는 과정**이라는 점을 체감함
    
- ts-node처럼 개발 효율성을 높여주는 도구의 존재가 큰 도움이 됨
- target/module 변경에 따라 컴파일된 JS가 얼마나 달라지는지 직접 보며 이해가 확실해짐
- 다른 프로젝트를 설정할 때도 이제 스스로 tsconfig를 구성할 수 있을 것 같은 자신감 생성됨

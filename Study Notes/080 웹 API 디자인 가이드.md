## 📝 핵심 개념 요약

- API 디자인 가이드라인은 “우리 팀의 API 설계 원칙을 통일하기 위한 문서”로,
API가 지속적으로 발전하고 변경되는 과정에서도 **일관성, 확장성, 예측 가능성**을 유지하도록 돕는 기준입니다.
이는 단순한 규칙이 아니라, 팀 전체의 **디자인 철학과 협업 언어**입니다.

## 📊 핵심 요약 표

| 구분 | 내용 요약 |
| --- | --- |
| **API 변화 이유** | 기능 추가, 삭제, 수정, 보안/성능 개선 등 지속적 변화 |
| **가이드라인 정의** | API 설계의 원칙·규칙을 문서화한 약속 |
| **목적** | 일관성 유지, 개발 효율 향상, 사용자 경험 개선, 미래 확장 대비 |
| **주요 구성 요소** | URI 규칙 / HTTP 메소드 원칙 / 응답 표준화 / 버전 관리 전략 |
| **대표 기업 사례** | Microsoft(엔터프라이즈 중심), Google(확장성 중심), Heroku(개발자 경험 중심) |
| **개발자 관점 장점** | 사용자: 예측 가능성 향상 / 제작자: 빠른 결정, 일관된 협업 가능 |
| **핵심 요약** | 좋은 API는 문서화된 일관성을 가진다. 가이드라인은 팀의 품질 기준이자 약속이다. |

### 💻 실습 내용 정리

### ✅ API 디자인 가이드라인 구성 예시

1. **URI 설계 규칙**
    - 복수형 명사 사용: `/users`, `/products`
    - 계층 구조 표현: `/users/{id}/orders`
    - 필터·정렬·페이지네이션 규칙 명시
2. **HTTP 메소드 원칙**
    
    
    | 메소드 | 용도 |
    | --- | --- |
    | GET | 리소스 조회 |
    | POST | 리소스 생성 |
    | PUT / PATCH | 리소스 수정 |
    | DELETE | 리소스 삭제 |
3. **응답 형식 표준화**
    - 일관된 구조 유지
    - 상태 코드 명시 (200, 201, 400, 404 등)
    - 오류 메시지 형식 통일
4. **버전 관리**
    - `/v1/users`, `Accept: application/vnd.api.v2+json`, `?version=2` 등 다양한 방식 지원

### ❗ 헷갈렸던 점 / 문제 해결:

**문제:**

- 각 기업의 가이드라인이 달라서 혼란
- URI를 단수로 써야 하는지 복수로 써야 하는
- 버전 명시 방법 중 어떤 게 가장 좋은가

**해결:**

- 공통적으로 지향하는 “일관성·명확성·예측 가능성” 중심으로 정리
- REST 원칙상 **리소스 컬렉션을 나타내므로 복수형 명사 사용**이 원칙
- 규모·환경에 따라 다르지만, 일반적으로 **URI 기반 버저닝(`/v1/`)**이 가장 명확

### 💡 느낀 점 / 배운 점:

- API 디자인은 단순히 작동하는 코드가 아니라 **사용자 경험과 협업의 언어**라는 걸 깨달았다.
- 기업별 가이드라인을 참고하면 실무적 감각을 익힐 수 있고,
    
    특히 “우리 팀의 표준 문서”를 만드는 것이 얼마나 중요한지 실감했다.
    
- 앞으로 API를 만들 때마다 **URI 구조, 응답 포맷, 버전 전략을 먼저 문서화**해보려고 한다.

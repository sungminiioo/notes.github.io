## 📝 핵심 개념 요약

- 브라우저와 Node.js는 **JS만 실행 가능** → TS는 그대로 실행 불가
- TS는 실행 전에 반드시 **TSC(타입스크립트 컴파일러)**가 JS로 변환(tranpile)
- 컴파일(compile) vs 트랜스파일(transpile) 차이 이해
- TS의 핵심 목적 = **타입 검사 + 실행 가능한 JS로 변환**

## 📊 핵심 요약 표

| 개념 | 설명 |
| --- | --- |
| TS → JS | 타입스크립트는 직접 실행 불가, 반드시 JS로 변환 필요 |
| TSC | 타입 검사 + 실행 가능한 자바스크립트 코드 생성 |
| Compile | 고급언어 → 저수준 언어(기계어)로 변환 |
| Transpile | 비슷한 수준의 언어끼리 변환 (TS → JS) |
| 런타임 | 브라우저/Node.js: JS만 실행 가능 |

### 💻 실습 내용 정리

### 1) 타입스크립트 파일 생성

```tsx
// index.ts
const message: string = "Hello TS";
console.log(message);
```

### 2) TSC로 JS 변환

```bash
tsc index.ts
```

### 3) 컴파일 결과

```jsx
// index.js
var message = "Hello TS";
console.log(message);
```

### 4) 실행 (Node.js)

```bash
node index.js
```

### ❗ 헷갈렸던 점 / 문제 해결:

### 헷갈렸던 점

- TS를 "컴파일"한다는 표현이 어떤 의미인지,
    
    실제로 기계어로 바꾸는 건지, JS로 바꾸는 건지 모호했음
    
- compile / transpile 구분이 혼동됨

### 문제 해결

- TS는 실제로 **트랜스파일 과정**을 거쳐 JS로 변환된다는 것을 명확히 이해
- compile은 보통 “기계어” 느낌, transpile은 “언어 사이 변환”
- 그림과 실행 흐름(TS → TSC → JS → 런타임)을 보며 구조적으로 이해됨

### 💡 느낀 점 / 배운 점:

- TS는 단순 타입 시스템이 아니라 **코드를 실행 가능한 JS로 바꾸는 빌드 단계**가 포함된 언어라는 점이 명확해짐
- 런타임은 TS를 전혀 이해하지 못하기 때문에,
    
    **TS는 개발자 경험용 언어고, JS가 최종 제품의 언어**라는 개념이 확실히 잡힘
    
- 앞으로 TS 기반 프로젝트에서는 tsconfig와 빌드 과정의 중요성이 더 크게 느껴짐

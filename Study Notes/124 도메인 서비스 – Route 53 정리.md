## 📝 핵심 개념 요약

- DNS는 인터넷의 전화번호부 역할을 하며 **도메인 → IP 주소 변환**
- Route 53은 AWS의 DNS 서비스 (이름은 포트 53에서 유래)
- 프론트 → 백엔드 통신 시 **HTTPS 적용을 위해 도메인이 반드시 필요**
- 도메인은 AWS, 가비아 등 다양한 곳에서 구매 가능 (Route 53만 고집 X)
- Route 53 주요 기능: 도메인 등록, DNS 레코드 관리, 헬스체크
- 실습: EC2 생성 → 프로젝트 클론 → DNS 연결 → A 레코드 추가 후 브라우저 접속

## 📊 핵심 요약 표

| 항목 | 요약 |
| --- | --- |
| DNS | 도메인 이름을 IP 주소로 변환하는 시스템 (전화번호부 역할) |
| Route 53 | AWS DNS 서비스. 도메인 등록·레코드 관리·헬스 체크 제공 |
| 왜 도메인이 필요한가 | HTTPS 적용, 프론트-백 간 안정적 통신, 사용자 친화성 |
| 도메인 구매처 | AWS, 가비아, 후이즈 등 (원하는 TLD에 따라 선택) |
| 실습 핵심 | EC2 생성 → Node 설치 → Git clone → Route 53 레코드 생성 → 가비아 네임서버 변경 → A 레코드에 IP 연결 |

### 💻 실습 내용 정리

### ✔ 1. EC2 인스턴스 생성

- 보안 그룹에서 **3000번 포트 인바운드 허용**

### ✔ 2. Node.js 설치

```bash
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
source ~/.bashrc
nvm install --lts
```

### ✔ 3. Git 설치

```bash
sudo dnf install git -y
```

### ✔ 4. 프로젝트 클론

```bash
git clone https://github.com/rjc1704/route53-aws-practice.git
cd route53-aws-practice/
```

### ✔ 5. Route 53 호스팅 영역 생성

- 도메인 입력 후 생성
- 기본 NS / SOA 레코드 자동 제공

### ✔ 6. NS 레코드 → 가비아 도메인 네임서버에 입력

- 뒤에 붙은 `.` 제거 후 값만 입력

### ✔ 7. A 레코드 생성

- 레코드 타입: **A**
- 값: **EC2 퍼블릭 IP**
- 서브도메인 원하는 경우 이름에 입력 (예: api)

### ✔ 8. 브라우저 확인

- 포트 3000으로 접속
    
    예)
    

```
http://api.wowthisplace.com:3000
```

### ❗ 헷갈렸던 점 / 문제 해결:

### ❓ 왜 IP 주소만으로 HTTPS 적용이 어려울까?

- HTTPS는 인증서가 도메인 기반으로 발급되기 때문
- 인증서의 CN(Common Name)에 도메인명이 필요함

### ❓ Route 53을 꼭 써야 하나?

- ❌ 절대 아님
- 도메인 구매처(TLD 종류)에 따라 **가비아·후이즈**가 더 나을 수도 있음

### ❓ NS 값 입력 후 적용이 늦는 이유?

- DNS 전파 시간: 최대 1~48시간
- 보통 10분~1시간 내 반영

### 💡 느낀 점 / 배운 점:

- 도메인과 DNS 구조를 알게 되니 “왜 HTTPS는 IP로 안 되는지”가 정확히 이해됨.
- Route 53이 단순 DNS 서비스가 아니라, 헬스 체크·트래픽 관리까지 가능한 고급 서비스라는 것도 새로 배움.
- 도메인 연결 과정이 생각보다 복잡하지만 흐름을 정확히 이해하니 앞으로는 쉽게 세팅할 수 있을 것 같다.

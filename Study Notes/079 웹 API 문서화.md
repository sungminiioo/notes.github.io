## 📝 핵심 개념 요약

- 좋은 API 문서는 개발자 간의 **협업 효율을 높이고**, **사용성을 향상시키며**, **유지보수 비용을 줄여주는 핵심 도구**입니다.
REST API 설계 시, 엔드포인트 구조, 요청/응답 형식, 상태 코드, 인증 방식 등을 명확히 정의하고 자동화 도구(Swagger, Postman 등)를 활용하면 문서 품질과 생산성이 함께 향상됩니다.

## 📊 핵심 요약 표

| 구분 | 내용 요약 |
| --- | --- |
| **API 문서의 필요성** | 개발자 간 협업, 오해 감소, 빠른 이해 가능 |
| **필수 구성 요소** | 엔드포인트 정보 / 요청 파라미터 / 응답 형식 / 인증 방식 / 코드 예시 |
| **좋은 문서의 특징** | 간결하고 구체적이며, 실제 예시를 포함 |
| **도구 활용** | Swagger, Redocly, Postman 등으로 자동화 문서 생성 |
| **핵심 포인트** | 사용자 입장으로 작성, 최신 상태 유지, 예제 코드 제공 |

### 💻 실습 내용 정리

### ✅ 엔드포인트 예시

```bash
POST /members - 새로운 회원 등록
GET /members - 회원 목록 조회
```

### ✅ 요청 파라미터 예시

```json
{
  "name": "Emily",
  "active": false  
}
```

| 필드 | 타입 | 필수 | 설명 |
| --- | --- | --- | --- |
| name | string | ✅ | 회원 이름 |
| active | boolean | ❌ | 기본값 true, 활성화 상태 |

### ✅ 응답 예시

```json
// 성공 (200 OK)
{
  "id": 2,
  "name": "Emily",
  "active": false
}
```

```json
// 실패 (400 Bad Request)
{
  "message": "field 'name' is required."
}
```

### ✅ 인증 예시

```bash
GET /users
Authorization: Bearer {JWT_TOKEN}
```

### ✅ Swagger 사용 예시

- Swagger UI에서 “Try it out”으로 API 직접 테스트 가능
- OpenAPI 명세 기반 자동 문서화

### ❗ 헷갈렸던 점 / 문제 해결:

**문제:**

- 요청·응답 예시를 어디까지 상세히 써야 하는가?
- 파라미터 설명 표기 형
- API 문서 자동화 도구의 차이

**해결:**

- 최소한 “성공/실패/에러” 케이스는 모두 명시
- 이름 / 타입 / 필수 여부 / 설명 / 기본값 순으로 정리
- Swagger는 주석 기반 자동화, Postman은 테스트 중심 공유에 강점

### 💡 느낀 점 / 배운 점:

- 문서화는 단순한 기록이 아니라 **개발자 간 언어를 통일하는 과정**임을 깨달았다.
- 좋은 API 문서는 “설명서”가 아니라 “대화의 기준점” 역할을 한다.
- Swagger와 Postman을 활용하면 문서와 테스트를 동시에 관리할 수 있어 매우 효율적이었다.
